# 记忆机制整理

> 本文整理自 AI Coding 知识库中关于 Claude Code 记忆机制的内容，涵盖 `CLAUDE.md`（手动记忆）与 `Auto Memory`（自动记忆）两种核心机制。

---

## 一、概述

在 Claude Code 中，每次会话都会从一个全新的上下文窗口开始。想要跨对话保留信息，主要依赖两种机制：

- **CLAUDE.md**：由用户手动编写，提供稳定、可复用的项目上下文（编码规范、工作流程、项目架构、行为约束）。
- **Auto Memory**：自动记忆，由 Claude Code 在交互过程中自动生成，记录在协作中逐步学习到的偏好、经验和模式。

这两类内容都会在新会话开始时加载，它们**本质上都属于"上下文"，而非绝对强制的配置项**。内容越明确、具体、精炼，执行效果越稳定。

两者的定位差异：

| 维度 | CLAUDE.md | Auto Memory |
|------|-----------|-------------|
| 核心定位 | **事先约定** | **在使用中学习** |
| 适合内容 | 长期、明确、需主动约束行为的规则 | 多次协作中逐步形成的知识 |
| 典型示例 | 团队规范、目录约定、测试流程 | 常见构建命令、调试经验、个人偏好 |
| 维护方式 | 用户手动编写和维护 | AI 自动生成，用户可审计编辑 |

两者相辅相成：CLAUDE.md 保证基础行为一致，Auto Memory 让 Claude Code 在持续协作中不断适应项目和使用者习惯。

## 二、CLAUDE.md（手动记忆）

### 2.1 概念与作用

`CLAUDE.md` 是 Markdown 文件，用于为 Claude Code 提供跨会话的持久指令。每次会话开始前，Claude Code 读取这些内容注入上下文，用于理解项目约束、工作流程和预期行为。

适合放入 CLAUDE.md 的内容：
- 构建、测试、发布等常用命令
- 代码风格、命名规则、目录约束等项目规范
- 项目结构、关键模块职责和重要依赖关系
- "始终执行 X"、"不要执行 Y" 这类全局规则

不适合放入的内容（应使用 Skill 或路径限定规则）：多步骤流程、只适用于代码库特定区域的内容。

### 2.2 使用场景

当出现以下情况时，考虑将内容加入 CLAUDE.md：
- Claude Code 犯了第二次同样的错误，需要固定规则避免重复
- 代码审查中发现 Claude Code 缺少本应了解的代码库背景信息
- 在本地对话中再次输入了和上次相同的纠正、说明和约定
- 新成员需要掌握同样的上下文才能高效参与项目

### 2.3 存放层级

| 作用范围 | 位置 | 用途 | 共享对象 |
|----------|------|------|----------|
| **托管策略** | macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md` | 组织级指令（编码标准、安全策略） | 组织内所有用户 |
| **项目指令** | `./CLAUDE.md` 或 `./.claude/CLAUDE.md` | 项目团队共享指令 | 通过源码管理共享 |
| **用户指令** | `~/.claude/CLAUDE.md` | 个人所有项目的偏好 | 仅本人，所有项目 |
| **本地指令** | `./CLAUDE.local.md` | 个人项目特定偏好 | 仅本人，当前项目 |

### 2.4 加载规则

- **向上查找**：从当前工作目录开始，沿目录树向上查找 `CLAUDE.md` 和 `CLAUDE.local.md`
- **拼接而非覆盖**：所有发现的文件都会被拼接进上下文
- **本地追加**：每个目录中 `CLAUDE.local.md` 追加在 `CLAUDE.md` 之后
- **子目录延迟加载**：子目录的 `CLAUDE.md` 不在启动时加载，仅在读取该子目录文件时加载
- **额外目录**：通过 `--add-dir` 添加的目录默认不加载记忆文件，需设置 `CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1`

### 2.5 创建与修改方式

1. **`/init` 命令**：自动生成初始 CLAUDE.md，分析代码库包含构建命令、测试说明和项目约定。已存在时会建议改进而非覆盖。
2. **`/memory` 命令**：查看和管理当前会话加载的持久指令和自动记忆内容，可打开编辑。
3. **`#` 操作符**：在对话中将一条内容作为持久指令直接写入。

### 2.6 编写规范

1. **控制文件大小**：建议每个 `CLAUDE.md` 控制在 **200 行以内**。过长会消耗更多上下文窗口，降低关键指令的遵循效果。
2. **使用清晰结构**：使用 Markdown 标题和列表分组，如：项目概览、构建和测试命令、代码风格、Git 工作流、重要约束。
3. **保持具体、可验证**：
   - 推荐："使用 2 个空格缩进"、"提交前运行 npm test"
   - 不推荐："正确格式化代码"、"测试你的改动"
4. **避免规则冲突**：定期检查根目录、子目录、`.claude/rules/` 中的规则，删除过期、重复、冲突内容。

### 2.7 导入路径（@path）

支持 `@path/to/import` 语法导入其他文件，被导入文件会在会话启动时与 CLAUDE.md 一起加载。

- 支持相对路径（相对于包含导入语句的文件）和绝对路径
- 可导入 README、package.json、Git 工作流说明、团队开发规范等
- 最多支持 **五层** 嵌套导入
- **注意**：导入改善组织方式，但**不减少上下文消耗**

### 2.8 CLAUDE.local.md

适合存放个人专用指令，应加入 `.gitignore`：
- 个人常用命令、本地开发环境差异
- 个人偏好的调试方式、不适合团队共享的临时说明

**注意**：被 Git 忽略的 `CLAUDE.local.md` 只在创建它的 worktree 中存在，不会自动同步到其他 worktree。如需多 worktree 共享，可从 `~/.claude/` 导入。

### 2.9 AGENTS.md 兼容

Claude Code 默认读取 `CLAUDE.md`，但可通过导入实现与 AGENTS.md 共享：

```markdown
@AGENTS.md

## Claude Code

Use plan mode for changes under `src/billing/`.
```

### 2.10 .claude/rules/ 路径限定规则

当项目规模扩大时，使用 `.claude/rules/` 拆分规则：

- **无条件加载的规则**：没有 `paths` frontmatter 的规则，启动时直接加载，等同于拆分版全局指令
- **路径特定规则**：使用 YAML frontmatter 的 `paths` 字段限定，只有处理匹配路径的文件时才加载

```yaml
---
paths:
  - "src/api/**/*.ts"
---

# API 开发规则
- 所有 API 接口都必须包含输入校验
```

- 支持符号链接共享规则（`ln -s`）
- 支持用户级规则 `~/.claude/rules/`，用户级先于项目级加载，项目级优先

### 2.11 大型团队管理

- **组织级部署**：放置在系统托管策略路径，**不能被个人设置排除**
- **排除特定文件**：使用 `claudeMdExcludes`（glob 语法，匹配绝对路径），在 `.claude/settings.local.json` 中配置。但**托管策略中的 CLAUDE.md 不能被排除**

## 三、Auto Memory（自动记忆）

### 3.1 概念

自动记忆让 Claude Code 在不需要手动维护的情况下，**跨会话积累项目相关知识**。它会根据内容是否值得长期保留，自行决定是否记录信息（构建命令、调试经验、架构说明、代码风格偏好、工作流习惯）。

### 3.2 启用与禁用

- **默认开启**
- 会话中输入 `/memory`，通过界面开关控制
- 项目设置：`"autoMemoryEnabled": false`
- 环境变量：`CLAUDE_CODE_DISABLE_AUTO_MEMORY=1`

### 3.3 存储位置

```
~/.claude/projects/<project>/memory/
```

- `<project>` 根据 Git 仓库路径推导
- 同一仓库的所有 worktree 和子目录默认共享同一个自动记忆目录
- 非 Git 目录使用项目根目录确定记忆位置
- 可配置 `autoMemoryDirectory` 自定义位置（不能来自项目设置，避免共享项目重定向到不安全位置）

### 3.4 目录结构

```
~/.claude/projects/<project>/memory/
├── MEMORY.md        # 索引文件
├── debugging.md     # 调试经验
├── api-conventions.md
└── ...
```

- `MEMORY.md`：索引文件，记录记忆内容的组织形式
- 主题文件：按主题拆分的详细笔记，按需读取

### 3.5 加载策略

- 每次对话开始前，只加载 `MEMORY.md` 的前 **200 行** 或前 **25 KB**（取较小者）
- 主题文件（如 `debugging.md`）**不在启动时加载**，仅在 Claude Code 认为相关时按需读取
- 会话过程中 Claude 会持续读写记忆文件，界面中可能看到 "Writing memory" 或 "Recalled memory"

### 3.6 审计与编辑

自动记忆文件本质是普通 Markdown，可随时：
- 手动查看
- 直接编辑
- 删除不想保留的内容

使用 `/memory` 可在会话中查看这些文件。

## 四、排查记忆相关问题

### 4.1 Claude Code 没有遵循 CLAUDE.md

**关键认知**：`CLAUDE.md` 内容不是系统提示的一部分，而是作为额外用户消息传递。Claude Code 会尽量遵循，但不等同于硬性强制执行。

排查顺序：
1. 运行 `/memory`，确认当前会话确实加载了 CLAUDE.md
2. 检查文件是否放在当前会话能加载的位置
3. 把指令写得具体、明确、可验证
4. 检查不同 CLAUDE.md 文件之间是否存在冲突

如需更强优先级，可使用 `--append-system-prompt`（更适合脚本或自动化场景）。

### 4.2 不知道自动记忆保存了什么

直接运行 `/memory`，打开自动记忆文件夹查看。

### 4.3 CLAUDE.md 太大

- 使用路径限定规则，让某些指令只在处理特定文件时加载
- 删除不需要每次会话都出现的内容
- 把全局规则、局部规则和任务说明拆开管理
- **注意**：`@path` 导入不减少上下文占用，只解决可维护性问题

### 4.4 /compact 之后指令似乎丢失

- 项目根目录的 `CLAUDE.md`：压缩后会重新从磁盘读取并注入
- 子目录的嵌套 `CLAUDE.md`：不会自动重新注入，只有后续再次读取对应子目录文件时才重新加载
- **可靠做法**：把需要保留的规则写进 `CLAUDE.md`，而不是只停留在对话内容里

## 五、记忆系统的通用设计原则

1. **上下文 vs 配置**：记忆本质上是"上下文"而非"强制配置"，效果取决于内容清晰度
2. **分层隔离**：组织/项目/个人/本地四级隔离，避免个人偏好污染团队规范
3. **路径限定**：复杂项目应使用 `.claude/rules/*.md`，让特定规则只在处理匹配文件时加载
4. **审计友好**：记忆文件是标准 Markdown，可随时查看、编辑、删除

## 六、不同工具的实现对比

| 特性 | Claude Code | Cursor | GitHub Copilot |
|------|-------------|--------|----------------|
| 手动记忆 | `CLAUDE.md` | `.cursorrules` | 通过 VS Code 设置 |
| 自动记忆 | `Auto Memory` | 无（需手动维护） | 无 |
| 路径限定规则 | `.claude/rules/*.md` | 无 | 无 |
| 层级加载 | 向上查找 + 拼接 | 单文件 | 编辑器配置 |
