# Subagent（子代理）笔记

> 子代理（Subagent）是专门处理特定任务的 AI 助手，拥有独立的上下文窗口、自定义系统提示词和特定的工具权限。当主代理判断任务符合某个子代理的描述时，会自动将任务委托给它。

---

## 一、为什么需要子代理

在使用 Claude Code 的过程中，主对话的上下文窗口是有限的。如果所有操作都在主对话中完成，很快会遇到以下问题：

- **上下文污染**：运行测试、分析日志等操作会产生大量输出，挤占主对话的上下文空间
- **权限难以控制**：某些任务需要限制工具使用（如只读审查），但主对话难以动态切换
- **角色混杂**：主对话既要负责协调，又要负责执行，容易顾此失彼

子代理的本质是**把独立任务派出去，让专家处理，主对话只接收结果**。这和现实中"项目经理把专项任务外包给专业团队"是同一个逻辑。

---

## 二、子代理的五大优势

1. **上下文保护**
   将探索和实现过程隔离到独立上下文，避免污染主对话。冗长的中间输出留在子代理内部，主对话只接收摘要和结论。

2. **权限隔离**
   通过限制可用工具强制执行约束。例如创建一个只读代码审查代理，它就无法修改任何文件。

3. **专业化**
   每个子代理针对特定领域深度定制。可以有一个专门做性能优化的代理，另一个专门做安全审计。

4. **成本控制**
   简单任务可路由到更快、更便宜的模型（如 Haiku），复杂任务才用 Sonnet/Opus。

5. **跨项目复用**
   用户级子代理放在 `~/.claude/agents/`，可在多个项目中共享配置。

## 三、内置子代理一览

Claude Code 内置了若干子代理，会在合适的场景自动调用：

| 子代理 | 模型 | 工具限制 | 典型场景 |
|--------|------|----------|----------|
| **Explore** | Haiku（速度快） | 只读工具，禁止 Write/Edit | 代码库搜索、文件发现、结构分析 |
| **Plan** | 继承主对话 | 只读工具，禁止 Write/Edit | plan mode 下的代码库研究 |
| **General-purpose** | 继承主对话 | 所有工具 | 复杂多步骤任务，需要同时探索和修改 |
| **statusline-setup** | Sonnet | 按任务分配 | 运行 `/statusline` 配置状态栏 |
| **Claude Code Guide** | Haiku | 按任务分配 | 回答 Claude Code 功能相关问题 |

Explore 代理特别值得注意：当 Claude Code 需要理解代码库但不需要修改时，会自动委托给 Explore。这样探索结果不会进入主对话上下文，极大节省 token。

## 四、三种核心使用模式

### 模式一：隔离高输出量操作

将产生大量中间内容的任务委托给子代理，主对话只接收摘要：

```text
使用一个子代理运行测试套件，并只报告失败的测试及其错误信息。
```

适用场景：
- 运行测试套件
- 分析日志文件
- 扫描大型代码库
- 获取大量文档

### 模式二：并行研究

多个调查任务彼此独立时，同时启动多个子代理并行处理：

```text
使用不同的子代理并行研究认证模块、数据库模块和 API 模块。
```

注意事项：
- 适合模块间依赖少、调查路径独立的场景
- 子代理完成后结果会返回主对话，如果同时运行很多且每个返回都很详细，主对话上下文仍可能吃紧

### 模式三：串联工作流

多步骤任务按顺序调用多个子代理，形成流水线：

```text
先使用 code-reviewer 查找性能问题，然后使用 optimizer 修复这些问题。
```

适用场景：
- 先分析、再修改
- 先研究、再实现
- 先审查、再优化

## 五、创建子代理的两种方式

### 方式一：通过 /agents 命令（推荐）

在 Claude Code 会话中执行 `/agents`，会打开带标签页的管理界面：

1. 切换到 `Library` 标签页，选择 `Create new agent`
2. 选择保存位置（Project / User / 其他）
3. 描述该子代理的职责和触发时机
4. 选择可用工具（白名单）或禁用工具（黑名单）
5. 选择模型（sonnet / opus / haiku / inherit）
6. 设置背景色（方便在会话中识别子代理运行状态）
7. 配置持久记忆作用域（user / project / local / None）
8. 保存并试用

### 方式二：手动编写子代理文件

子代理文件是 **YAML frontmatter + Markdown system prompt** 格式的 Markdown 文件：

```markdown
---
name: code-review
description: 资深代码审查专家。在代码更改后主动使用。
tools: Read, Glob, Grep, Bash
model: sonnet
permissionMode: default
memory: project
---

你是一名代码审查员。被调用时，请分析代码，并针对代码质量、安全性和最佳实践提供具体、可执行的反馈。
```

文件存放位置决定作用域和优先级：

| 优先级 | 位置 | 范围 | 说明 |
|--------|------|------|------|
| 1（最高） | 托管设置 | 组织范围 | 管理员统一部署 |
| 2 | `--agents` CLI 标志 | 当前会话 | 启动时传入 JSON，不保存到磁盘 |
| 3 | `.claude/agents/` | 当前项目 | 可提交到版本控制，团队共享 |
| 4 | `~/.claude/agents` | 个人所有项目 | 跨项目复用 |
| 5（最低） | 插件的 agents 目录 | 启用该插件的位置 | 随插件安装 |

> 手动添加文件后，需要**重启会话或使用 `/agents` 立即加载**。

## 六、子代理配置字段详解

### 必填字段

| 字段 | 说明 |
|------|------|
| `name` | 唯一标识符，使用小写字母和连字符 |
| `description` | 触发条件描述——"在什么情况下委托给该子代理" |

### 常用可选字段

| 字段 | 说明 |
|------|------|
| `tools` | 白名单工具列表，省略则继承所有工具 |
| `disallowedTools` | 黑名单工具，从继承列表中移除 |
| `model` | `sonnet` / `opus` / `haiku` / `inherit`（默认） |
| `permissionMode` | `default` / `acceptEdits` / `plan` / `bypassPermissions` 等 |
| `mcpServers` | 可用的 MCP 服务器（可内联定义或引用已有配置） |
| `skills` | 预加载的技能（完整内容注入子代理上下文） |
| `memory` | 持久记忆作用域：`user` / `project` / `local` |
| `isolation` | 设置为 `worktree` 时在隔离 git worktree 中运行 |
| `background` | 设置为 `true` 时作为后台任务运行 |
| `maxTurns` | 子代理停止前允许的最大思考+行动轮数 |
| `color` | 在任务列表和记录中的显示颜色 |

### 权限模式说明

| 模式 | 行为 |
|------|------|
| `default` | 标准权限检查，危险操作前提示 |
| `acceptEdits` | 自动接受工作目录内的文件编辑 |
| `auto` | 后台分类器检查命令和受保护目录写入 |
| `dontAsk` | 自动拒绝权限提示；显式允许的工具仍可工作 |
| `bypassPermissions` | 跳过权限提示（Yolo 模式） |
| `plan` | 只读探索，给出方案但不修改 |

> 父级使用 `bypassPermissions` 或 `acceptEdits` 时，父级设置优先，子代理无法覆盖。

## 七、控制子代理的能力边界

### 1. 工具白名单与黑名单

```yaml
# 白名单：只允许使用指定工具
tools: Read, Grep, Glob, Bash

# 黑名单：继承所有工具，但排除指定工具
disallowedTools: Write, Edit
```

如果同时设置，处理顺序是：继承全部 -> 删除黑名单 -> 用白名单筛选。某个工具同时在两边，最终被禁用。

### 2. 限制子代理生成子代理

当代理通过 `claude --agent` 作为主会话代理运行时，可用 `Agent(agent_type)` 语法限制它能生成哪些子代理：

```yaml
tools: Agent(worker, researcher), Read, Bash
```

完全省略 `Agent` 则该代理不能生成任何子代理。

### 3. MCP 服务器限定到子代理

```yaml
mcpServers:
  # 内联定义：仅该子代理可用，主对话看不到
  - playwright:
      type: stdio
      command: npx
      args: ["-y", "@playwright/mcp@latest"]
  # 字符串引用：复用主对话已配置的服务器
  - github
```

内联定义的服务器在子代理启动时连接、结束时断开，字符串引用则共享父会话连接。

### 4. 使用钩子定义条件规则

`PreToolUse` 钩子可在工具执行前运行验证脚本，实现更细粒度的控制：

```yaml
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: "command"
          command: "./scripts/validate-readonly-query.sh"
```

例如验证数据库查询是否为只读 SELECT，如果是写入操作则拦截。

### 5. 禁用特定子代理

在 `settings.json` 中：

```json
{
  "permissions": {
    "deny": ["Agent(Explore)", "Agent(my-custom-agent)"]
  }
}
```

或通过 CLI 标志：

```bash
claude --disallowedTools "Agent(Explore)"
```

## 八、持久记忆：让子代理越用越聪明

`memory` 字段给子代理一个可跨对话保留的持久记忆目录，随时间积累知识：

| 作用域 | 位置 | 适用场景 |
|--------|------|----------|
| `user` | `~/.claude/agent-memory/<name>/` | 知识跨项目通用 |
| `project` | `.claude/agent-memory/<name>/` | 知识与项目相关，可版本控制共享 |
| `local` | `.claude/agent-memory-local/<name>/` | 项目相关但不提交到版本控制 |

启用后：
- 子代理系统提示词会包含读写记忆目录的说明
- 自动加载 `MEMORY.md` 的前 200 行或 25KB
- Read、Write、Edit 工具自动启用，方便子代理管理记忆文件

**使用建议**：
- 可以要求子代理开始工作前先查阅记忆："请审查这个 PR，并检查你的记忆中是否有之前见过的相关模式"
- 可以要求子代理完成后更新记忆："把学到的内容保存到你的记忆中"
- 也可在子代理文件中直接加入记忆维护指令，让它主动积累团队级知识

## 九、主对话 vs 子代理：如何选择

| 场景 | 推荐方式 | 原因 |
|------|----------|------|
| 需要频繁沟通、调整方向 | 主对话 | 上下文连续，反馈即时 |
| 小且明确的修改 | 主对话 | 启动子代理有额外开销 |
| 希望响应更快 | 主对话 | 省去委托和返回的交互 |
| 运行测试/分析日志/扫描代码库 | 子代理 | 隔离高输出，保护主上下文 |
| 需要限制工具权限 | 子代理 | 白名单/黑名单精确控制 |
| 可独立完成的任务 | 子代理 | 派出去做，只要结果 |
| 需要专业角色处理 | 子代理 | 深度定制的系统提示词 |

简单判断：**如果任务可以"派出去做"，最后只要一个结果，就适合用子代理。**

## 十、Superpowers 框架中的子代理

在 [[Superpowers]] 框架中，子代理是核心执行单元。其 7 阶段开发流程的第 4 步"子代理驱动开发"明确要求：

> 为每个任务派遣子代理，进行两阶段审查（规格合规性 + 代码质量）。

Superpowers 还提供了专门的协作技能：
- **派遣并行代理**：并发子代理工作流
- **子代理驱动开发**：通过两阶段审查快速迭代
- **使用 Git 工作树**：配合 `isolation: worktree` 实现完全隔离的开发分支

这体现了子代理从"辅助功能"到"核心架构"的演进。
