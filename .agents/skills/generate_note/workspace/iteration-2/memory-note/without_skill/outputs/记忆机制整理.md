# 一、概述

在 `Claude Code` 中，每次会话都会从一个全新的上下文窗口开始，想要跨对话保留信息，主要依赖两种机制：`CLAUDE.md` 和 `auto memory`。

| 机制 | 定位 | 核心特征 |
|------|------|----------|
| `CLAUDE.md` | 手动记忆 | 由用户手动编写，提供稳定、可复用的项目上下文，例如编码规范、工作流程、项目架构和行为约束 |
| `auto memory` | 自动记忆 | 由 `Claude Code` 在交互过程中自动生成，记录协作中逐步学习到的偏好、经验和模式 |

> **关键认知**：这两类内容都会在新会话开始时加载，它们**本质都属于"上下文"，而非绝对强制的配置项**。因此，内容越明确、越具体、越精炼，`Claude Code` 的执行效果通常越稳定。

两者的定位不同：
- `CLAUDE.md` 强调"**事先约定**"，更适合承载长期、明确、需要主动约束 `Claude Code` 行为的规则，例如团队规范、目录约定、测试流程和修改边界。
- `auto memory` 强调"**在使用中学习**"，更适合沉淀在多次协作中逐步形成的知识，例如常见构建命令、调试经验、个人偏好，以及 `Claude Code` 从更正中总结出来的工作模式。

`CLAUDE.md` 和 `auto memory` 是**相辅相成**的，两者结合后，既可以保证基础行为一致，也能让 `Claude Code` 在持续协作中不断适应具体项目和使用者习惯。


# 二、CLAUDE.md

## 1、概念

`CLAUDE.md` 是 `Markdown` 文件，用于为 `Claude Code` 提供跨会话的持久指令。文件内容由用户用纯文本形式编写，`Claude Code` 在每次会话开始前读取这些内容注入上下文，用于理解项目约束、工作流程和预期行为等。

`CLAUDE.md` 既可以服务于单个具体项目，也可以服务于个人工作流，或在组织层面统一生效。

`CLAUDE.md` 应保留 `Claude Code` 在每次对话中都应该默认掌握的事实和规则，例如：
- 构建、测试、发布等常用命令
- 代码风格、命名规则、目录约束等项目规范
- 项目结构、关键模块职责和重要依赖关系
- "始终执行 X"、"不要执行 Y" 这类全局规则

> 如果某条内容属于多步骤流程，或者只适用于代码库中的某个特定区域，则不建议放入 `CLAUDE.md`，这类内容更推荐放到 `skill` 或路径限定规则中。

## 2、使用 CLAUDE.md 的场景

可以将 `CLAUDE.md` 理解为项目的"长期上下文记录"。**它适合存放那些如果不写下来，就需要在每次对话中反复解释的信息**。

当出现以下情况时，可以考虑将相关内容加入 `CLAUDE.md`：
- `Claude Code` 犯了第二次同样的错误，需要通过固定规则避免重复发生
- 代码审查中发现，`Claude Code` 缺少本应了解的代码库背景信息
- 你在本地对话中再次输入了和上次相同的纠正、说明和约定
- 新加入的成员需要掌握同样的上下文，才能高效参与项目

## 3、存放位置

`CLAUDE.md` 文件可以放置在不同的位置，对应不同的层级范围：

| 作用范围 | 位置 | 用途 | 使用场景示例 | 共享对象 |
|----------|------|------|-------------|----------|
| **托管策略** | macOS：`/Library/Application Support/ClaudeCode/CLAUDE.md`<br>Linux 和 WSL：`/etc/claude-code/CLAUDE.md`<br>Windows：`C:\Program Files\ClaudeCode\CLAUDE.md` | 由 IT/DevOps 管理的组织级指令 | 公司编码标准、安全策略、合规要求 | 组织内所有用户 |
| **项目指令** | `./CLAUDE.md` 或 `./.claude/CLAUDE.md` | 项目团队共享的指令 | 项目架构、编码标准、常用工作流 | 通过源码管理共享给团队成员 |
| **用户指令** | `~/.claude/CLAUDE.md` | 适用于所有项目的个人偏好 | 代码风格偏好、个人工具快捷方式 | 仅你本人，适用于所有项目 |
| **本地指令** | `./CLAUDE.local.md` | 个人的项目特定偏好；**应加入 `.gitignore`** | 你的沙盒 URL、偏好的测试数据 | 仅你本人，当前项目 |

## 4、CLAUDE.md 的加载规则

`Claude Code` 会从当前工作目录开始，沿着目录树**向上查找** `CLAUDE.md` 文件，检查沿途每一个目录中的 `CLAUDE.md` 和 `CLAUDE.local.md` 文件。所有发现的文件都会被**拼接进上下文**，而不是相互覆盖。

- 在每个目录中，`CLAUDE.local.md` **追加**在同层 `CLAUDE.md` 之后
- 子目录的 `CLAUDE.md` 和 `CLAUDE.local.md` **不会在启动时加载**，而是在 `Claude Code` **读取子目录文件**时才被包含进来

`--add-dir` 标志可以让 `Claude Code` 访问主工作目录之外的额外目录，但默认情况下不会自动加载记忆文件。如需加载，需设置环境变量：

```shell
CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1 claude --add-dir ../shared-config
```

这会从额外目录加载 `CLAUDE.md`（或 `.claude/CLAUDE.md`）、`CLAUDE.local.md`、`.claude/rules/*.md`。

## 5、CLAUDE.md 注释

在 `CLAUDE.md` 中可以使用块级 `HTML` 注释，用于给人工维护者留下说明：

```html
<!-- 注释 -->
```

> 这些注释在注入到 `Claude Code` 的上下文时会被移除，因此不会消耗上下文 token。
>
> 注意：代码块内部的注释仍然会保留；`Claude Code` 使用 `Read` 工具直接打开某个 `CLAUDE.md` 文件时，这些注释仍然可见。

## 6、创建或修改 CLAUDE.md

### （1）项目级 CLAUDE.md 定位

项目级 `CLAUDE.md` 的存放位置可以是 `./CLAUDE.md` 或者 `.claude/CLAUDE.md`。它适合添加适用于所有项目参与者的指令：构建与测试命令、编码标准、架构决策以及常见工作流。

创建或修改 `CLAUDE.md` 有三种主要方式：

| 方式 | 命令/操作 | 说明 |
|------|----------|------|
| 初始化 | `/init` | 自动生成初始 `CLAUDE.md`，分析代码库并包含构建命令、测试说明和项目约定。若已存在则建议改进而非覆盖 |
| 管理 | `/memory` | 查看和管理当前会话加载的持久指令和自动记忆内容，是检查记忆是否正确加载和故障排查的主要入口 |
| 写入 | `#` 操作符 | 在对话中使用 `#` 操作符将一条内容作为持久指令写入 |

> 新版初始化流程：指定 `CLAUDE_CODE_NEW_INIT` 环境变量可启用更高级的交互式初始化。

### （2）使用 `/init` 自动生成示例

```markdown
# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 项目概况

Rust 项目，使用 Cargo 构建，Rust edition 2024。

## 构建与检查

```bash
cargo build  # 构建项目
cargo test   # 运行测试
cargo fmt    # 格式化代码（提交前必须运行）
cargo clippy # Lint 检查（提交前必须通过）
```

## 分支命名

使用 conventional 前缀：`feat/xxx`、`fix/xxx`、`chore/xxx`、`docs/xxx` 等。

## 提交信息

采用 Conventional Commits 格式：`feat:`、`fix:`、`chore:`、`docs:` 等。
```


# 三、编写规范的 CLAUDE.md

## 1、编写有效的指令

`CLAUDE.md` 会在每次会话开始时自动加载到上下文窗口中，并与对话内容一起消耗 token。它本质上是提供给 Claude 的上下文信息，而不是强制执行的配置文件。**因此，指令写得是否清晰，会直接影响 Claude 遵循指令的稳定性**。

编写 `CLAUDE.md` 时应遵循以下规则：

| 规则 | 说明 | 示例 |
|------|------|------|
| **控制文件大小** | 建议每个 `CLAUDE.md` 文件控制在 **200 行以内** | 文件过长会降低关键指令的遵循效果 |
| **使用清晰结构** | 使用 Markdown 标题和项目符号分组 | `# 项目概览`、`# 构建和测试命令`、`# 代码风格` |
| **保持具体、可验证** | 指令应当尽量具体，避免模糊描述 | 推荐："使用 2 个空格缩进"；不推荐："正确格式化代码" |
| **避免规则冲突** | 定期检查不同层级的 `CLAUDE.md` 是否存在矛盾 | 根目录、子目录、`.claude/rules/` 中的规则文件 |

> 如果内容确实很庞大，可通过**路径限定规则**（`.claude/rules/*.md`）让某些指令只在处理特定文件时加载。使用 `@path` 导入虽然改善组织方式，但**不会减少上下文消耗**。

## 2、导入路径

`CLAUDE.md` 支持使用 `@path/to/import` 语法导入其他文件。被导入的文件会被展开，并在会话启动时与 `CLAUDE.md` 一起加载到上下文中。

可导入的内容包括：README、package.json、Git 工作流说明、团队开发规范、个人偏好设置。

```markdown
请参阅 @README 了解项目概览，并参阅 @package.json 查看项目可用的 npm 命令。

# 补充说明

- Git 工作流 @docs/git-instructions.md
```

- 相对路径：相对于**包含导入语句的文件**解析
- 绝对路径：指向系统中的固定文件（跨项目不建议）
- 被导入的文件还可以继续导入其他文件，最多支持**五层**

## 3、CLAUDE.local.md

如果某些指令只适合你个人使用，不想提交到 Git 版本控制，可以在项目根目录创建 `CLAUDE.local.md`，并加入 `.gitignore`。

`CLAUDE.local.md` 会和 `CLAUDE.md` 一起加载，并以相同方式处理。

适合放入 `CLAUDE.local.md` 的内容：
- 个人常用命令
- 本地开发环境差异
- 个人偏好的调试方式
- 不适合团队共享的临时说明

> 注意：被 Git 忽略的 `CLAUDE.local.md` 只会存在于创建它的那个 worktree，不会自动同步到其他 worktree。如需共享，可从用户家目录的 `.claude` 中导入文件。

## 4、AGENTS.md 兼容

Claude Code 默认读取 `CLAUDE.md`，而不是 `AGENTS.md`。如果仓库已为其他编码代理维护了 `AGENTS.md`，不需要重复写一份。可以创建 `CLAUDE.md` 导入 `AGENTS.md`，让多个工具共享同一套基础指令：

```markdown
@AGENTS.md

## Claude Code

Use plan mode for changes under `src/billing/`.
```

## 5、使用 `.claude/rules/` 整理规则

### （1）为什么使用 `.claude/rules/`

当项目不断扩大时，所有规范堆积在一个 `CLAUDE.md` 文件里会出现几个问题：
- 内容越来越长，不利于阅读和维护
- 不同主题的规则混杂在一起，不便于团队协作更新
- 某些只适用于局部目录或特定类型文件的规则，被无差别加载到上下文中

**`.claude/rules/` 的作用就是把规则拆分为独立文件，按主题组织**。同时支持按文件路径生效，只有当 Claude 处理匹配路径的文件时，对应规则才会被加载，避免无关规则长期占用上下文。

### （2）基本设置方式

把 Markdown 文件放到项目的 `.claude/rules/` 目录中即可。每个文件最好只覆盖一个主题：

```shell
your-project/
├── .claude/
│   ├── CLAUDE.md           # 主项目指令
│   └── rules/
│       ├── code-style.md   # 代码风格指南
│       ├── testing.md      # 测试约定
│       └── security.md     # 安全要求
```

`.claude/rules/` 下的所有 `.md` 文件都会被递归发现，因此也可以继续按领域拆分子目录（如 `frontend/`、`backend/`）。

### （3）无条件加载的规则

如果规则文件没有写 `paths` 的 YAML frontmatter，它会在启动时直接加载，优先级与 `CLAUDE.md` 相同。适合放置对整个项目都成立的全局规范：通用代码风格、提交前测试要求、安全基线要求、全项目统一的命名约定。

### （4）路径特定规则

如果某些规则只适用于特定目录或文件类型，可以在规则文件顶部使用 YAML frontmatter 的 `paths` 字段进行限定：

```yaml
---
paths:
  - "src/api/**/*.ts"
---

# API 开发规则

- 所有 API 接口都必须包含输入校验
- 使用统一的错误响应格式
- 补充 OpenAPI 文档注释
```

`paths` 支持 glob 模式：

| 模式 | 含义 |
|------|------|
| `**/*.ts` | 匹配任意目录下的 TypeScript 文件 |
| `src/**/*` | 匹配 `src/` 目录下的所有文件 |
| `*.md` | 匹配项目根目录中的所有 Markdown 文件 |
| `src/**/*.{ts,tsx}` | 匹配 `src/` 下的 `.ts` 和 `.tsx` 文件 |

### （5）使用符号链接共享规则

`.claude/rules` 目录支持符号链接，可以把一套公共规则维护在统一位置，再链接到多个项目复用：

```shell
ln -s ~/shared-claude-rules .claude/rules/shared
ln -s ~/company-standards/security.md .claude/rules/security.md
```

### （6）项目级规则

除了项目内的 `.claude/rules/`，还可以在用户主目录下维护一套个人规则：

```shell
~/.claude/rules/
├── preferences.md    # 个人编码偏好
└── workflows.md      # 个人工作流习惯
```

这类规则会应用到所有项目中。加载顺序上，**用户级规则会先于项目规则加载**，如果二者发生冲突，通常以**项目规则优先**。


# 四、大型团队管理 CLAUDE.md

## 1、部署组织级 CLAUDE.md

在团队或组织内推广 `Claude Code` 时，可以通过部署一份集中管理的 `CLAUDE.md`，为所有开发者提供统一的行为指引。

组织级 `CLAUDE.md` 需要放置在系统指定的托管策略路径中：

| 系统 | 路径 |
|------|------|
| macOS | `/Library/Application Support/ClaudeCode/CLAUDE.md` |
| Linux / WSL | `/etc/claude-code/CLAUDE.md` |
| Windows | `C:\Program Files\ClaudeCode\CLAUDE.md` |

> **组织级 `CLAUDE.md` 不能被个人设置排除**，以确保组织级规范不会被用户本地配置绕过。

## 2、排除特定 CLAUDE.md 文件

在大型 monorepo 中，仓库上层目录或其他团队维护的目录里可能存在 `CLAUDE.md` 或 `.claude/rules/` 文件，这些指令不一定和你当前负责的部分有关，甚至可能对当前工作产生干扰。

可以使用 `claudeMdExcludes` 设置，按路径或 glob 模式排除特定的 `CLAUDE.md` 文件或规则目录。一般在 `.claude/settings.local.json` 中设置：

```json
{
  "claudeMdExcludes": [
    "**/monorepo/CLAUDE.md",
    "/home/user/monorepo/other-team/.claude/rules/**"
  ]
}
```

`claudeMdExcludes` 使用 **glob 语法**，针对**绝对文件路径**进行匹配。**托管策略中的 `CLAUDE.md` 不能被排除**。


# 五、自动记忆（Auto Memory）

## 1、概念

自动记忆（`Auto Memory`）可以让 `Claude Code` 在不需要手动维护的情况下，**跨会话积累项目相关知识**。它会在工作过程中，根据内容是否值得长期保留，自行决定是否记录一些信息，例如构建命令、调试经验、架构说明、代码风格偏好和工作流习惯。

> `Claude Code` 不会在每次会话后都写入记忆，而是**只保存那些对未来对话可能有帮助的内容**。

## 2、启用和禁用

自动记忆默认是**开启**的。可以通过两种方式控制：

| 方式 | 操作 |
|------|------|
| 界面开关 | 在会话中输入 `/memory`，通过界面里的自动记忆开关启用或关闭 |
| 配置项 | 在项目设置中设置 `autoMemoryEnabled` |
| 环境变量 | `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1` |

关闭自动记忆的配置示例：

```json
{
  "autoMemoryEnabled": false
}
```

## 3、存储位置

每个项目都有独立的自动记忆目录，默认位置是：

```bash
~/.claude/projects/<project>/memory/
```

其中 `<project>` 通常根据 Git 仓库路径推导得到，因此：
- 同一个仓库的所有 worktree
- 同一个仓库的所有子目录

默认共享同一个自动记忆目录。如果当前目录没有被 Git 纳入版本控制，则 `<project>` 使用项目根目录来确定记忆位置。

如需改存到其他地方，可在**用户设置**或**本地设置**中配置 `autoMemoryDirectory`：

```json
{
  "autoMemoryDirectory": "~/my-custom-memory-dir"
}
```

> 该配置可来自 policy 设置、local 设置、user 设置，但**不能来自项目设置**（`.claude/settings.json`），避免共享项目把自动记忆重定向到不安全的位置。

## 4、目录结构

自动记忆目录中通常会包含一个入口文件和若干主题文件：

```shell
~/.claude/projects/<project>/memory/
├── MEMORY.md        # 索引文件
├── debugging.md     # 调试经验
├── api-conventions.md
└── ...
```

| 文件 | 作用 |
|------|------|
| `MEMORY.md` | **索引文件**，记录记忆内容的组织形式 |
| `debugging.md`、`api-conventions.md` 等 | 按主题拆分的详细笔记 |

`Claude Code` 会根据需要继续创建其他主题文件。

## 5、工作方式

每次对话开始前，`Claude Code` **不会**把整个记忆目录全部读入上下文，而是只加载 `MEMORY.md` 的一部分：
- 前 200 行
- 或前 25 KB
- 取两者中较小者

超过这个范围的内容不会在启动时自动加载。为了保持启动上下文精简，`Claude Code` 会尽量把详细信息拆到单独的主题文件中，而让 `MEMORY.md` 保持简洁。

像 `debugging.md`、`patterns.md` 这类主题文件**不会在会话开始时自动加载**。只有当 `Claude Code` 认为相关信息有用时，才会按需读取这些文件。

在会话过程中，Claude 也会持续读写这些记忆文件。如果你在界面中看到：
- `Writing memory`
- `Recalled memory`

即表示自动记忆正在工作。

## 6、审计与编辑

自动记忆文件本质上就是普通的 `Markdown` 文件，因此你可以随时：
- 手动查看
- 直接编辑
- 删除不想保留的内容

如果想在会话中查看这些文件，可以使用 `/memory`。


# 六、排查记忆相关问题

## 1、Claude Code 没有遵循 CLAUDE.md

> **需要明确一点：`CLAUDE.md` 的内容并不是系统提示本身的一部分，而是会在系统提示之后，作为一条额外的用户消息传递给模型**。`Claude Code` 会读取并尽量遵循这些指令，但这并不等同于硬性强制执行。

排查顺序：

| 步骤 | 操作 | 目的 |
|------|------|------|
| 1 | 运行 `/memory` | 确认当前会话中确实加载了 `CLAUDE.md` 和 `CLAUDE.local.md` |
| 2 | 检查文件位置 | 确认 `CLAUDE.md` 放在了当前会话能加载的位置 |
| 3 | 优化指令质量 | 把指令写得具体、明确、可验证 |
| 4 | 检查冲突 | 排查不同 `CLAUDE.md` 文件之间是否存在矛盾规则 |

如果某条指令必须提升到**更强的优先级**，可以考虑使用 `--append-system-prompt`。这种方式更接近系统提示层级，但需要在每次调用时显式传入，更适合脚本或自动化场景。

另外，`InstructionsLoaded` hook 可以准确记录哪些指令文件被加载了、在什么时候加载、以及为什么会被加载，对于排查路径规则或子目录中延迟加载的指令尤其有用。

## 2、不知道自动记忆保存了什么

直接运行 `/memory`，然后打开自动记忆文件夹查看。

## 3、CLAUDE.md 太大

`CLAUDE.md` 过长会占用更多上下文空间，也可能降低 `Claude Code` 对关键指令的遵循效果。通常建议将单个 `CLAUDE.md` 控制在 200 行以内。

如果内容已经很多，可以考虑：
- 使用路径限定规则，让某些指令只在处理特定文件时加载
- 删除那些并不需要在每次会话中都出现的内容
- 把全局规则、局部规则和任务说明拆开管理

> 把内容拆成 `@path` 导入虽然有助于组织结构更清晰，但**不会减少上下文占用**。

## 4、`/compact` 之后指令似乎丢失了

执行 `/compact` 之后，并不是所有指令都会以同样方式保留下来：

| 指令来源 | `/compact` 后行为 |
|----------|-------------------|
| 项目根目录下的 `CLAUDE.md` | 压缩后**继续保留**，`Claude Code` 会重新从磁盘读取并再次注入 |
| 子目录中的嵌套 `CLAUDE.md` | **不会自动重新注入**，只有在后续再次读取对应子目录文件时才会重新加载 |
| 仅存在于对话中的指令 | **丢失**，因为压缩后对话历史被精简 |

如果希望某条规则在压缩后依然稳定保留，最可靠的方法是把它写进 `CLAUDE.md`，而不是只停留在对话内容里。


# 七、记忆系统的通用设计原则

综合 `CLAUDE.md` 和 `auto memory` 的设计，可以总结出以下通用原则：

| 原则 | 说明 |
|------|------|
| **上下文 vs 配置** | 记忆本质上是"上下文"而非"强制配置"，效果取决于内容清晰度 |
| **分层隔离** | 组织 / 项目 / 个人 / 本地四级隔离，避免个人偏好污染团队规范 |
| **路径限定** | 复杂项目应使用路径限定规则（`.claude/rules/*.md`），让特定规则只在处理匹配文件时加载 |
| **审计友好** | 记忆文件是标准 Markdown，可随时查看、编辑、删除 |
