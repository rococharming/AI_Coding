# 一、概述

在 Claude Code 中，每次会话都会从一个全新的上下文窗口开始，想要跨对话保留信息，主要依赖两种机制：**CLAUDE.md** 和 **Auto Memory**。

- **CLAUDE.md**：由用户手动编写，用于向 Claude Code 提供稳定、可复用的项目上下文，例如编码规范、工作流程、项目架构和行为约束。
- **Auto Memory**：自动记忆，由 Claude Code 在交互过程中自动生成，用于记录在协作中逐步学习到的偏好、经验和模式。

这两类内容都会在新会话开始时加载，它们**本质都属于"上下文"，而非绝对强制的配置项**。因此，内容越明确、越具体、越精炼，Claude Code 的执行效果通常越稳定。

> 两者相辅相成：CLAUDE.md 保证基础行为一致，Auto Memory 让 Claude Code 在持续协作中不断适应具体项目和使用者习惯。

# 二、CLAUDE.md

## 1、概念

CLAUDE.md 是 Markdown 文件，用于为 Claude Code 提供跨会话的持久指令。文件内容由用户用纯文本形式编写，Claude Code 在每次会话开始前读取这些内容注入上下文，用于理解项目约束、工作流程和预期行为等。

CLAUDE.md 既可以服务于单个具体项目，也可以服务于个人工作流，或在组织层面统一生效。应保留 Claude Code 在每次对话中都应该默认掌握的事实和规则，例如：

- 构建、测试、发布等常用命令
- 代码风格、命名规则、目录约束等项目规范
- 项目结构、关键模块职责和重要依赖关系
- "始终执行 X"、"不要执行 Y" 这类全局规则

## 2、存放位置

CLAUDE.md 文件可以放置在不同的位置，对应不同的层级范围：

| 作用范围 | 位置 | 用途 | 使用场景示例 | 共享对象 |
|------|------|------|--------|------|
| **托管策略** | macOS：`/Library/Application Support/ClaudeCode/CLAUDE.md`<br>Linux 和 WSL：`/etc/claude-code/CLAUDE.md`<br>Windows：`C:\Program Files\ClaudeCode\CLAUDE.md` | 由 IT/DevOps 管理的组织级指令 | 公司编码标准、安全策略、合规要求 | 组织内所有用户 |
| **项目指令** | `./CLAUDE.md` 或 `./.claude/CLAUDE.md` | 项目团队共享的指令 | 项目架构、编码标准、常用工作流 | 通过源码管理共享给团队成员 |
| **用户指令** | `~/.claude/CLAUDE.md` | 适用于所有项目的个人偏好 | 代码风格偏好、个人工具快捷方式 | 仅你本人，适用于所有项目 |
| **本地指令** | `./CLAUDE.local.md` | 个人的项目特定偏好；**应加入 `.gitignore`** | 你的沙盒 URL、偏好的测试数据 | 仅你本人，当前项目 |

## 3、加载规则

Claude Code 会从当前工作目录开始，沿着目录树**向上查找** CLAUDE.md 文件，检查沿途每一个目录中的 `CLAUDE.md` 和 `CLAUDE.local.md` 文件。所有发现的文件都会被**拼接**进上下文，而不是相互覆盖。

- 在每个目录中，`CLAUDE.local.md` 追加在 `CLAUDE.md` 之后
- 子目录的 `CLAUDE.md` **不在启动时加载**，仅在读取该子目录文件时被包含进来
- `--add-dir` 标志访问额外目录时，默认不自动加载记忆文件，需设置环境变量：

```shell
CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1 claude --add-dir ../shared-config
```

## 4、创建或修改方式

创建或修改 CLAUDE.md 有三种主要方式：

| 方式 | 命令 | 说明 |
|------|------|------|
| 初始化 | `/init` | 自动生成初始 CLAUDE.md，分析代码库包含构建命令、测试说明和项目约定 |
| 管理 | `/memory` | 查看和管理当前会话加载的持久指令和自动记忆内容 |
| 写入 | `#` 操作符 | 在对话中将一条内容作为持久指令直接写入 |

> 若已存在 CLAUDE.md，`/init` 会建议改进而非覆盖。设置 `CLAUDE_CODE_NEW_INIT=1` 可启用新版交互式初始化流程。

## 5、编写规范

### （1）控制文件大小

建议每个 CLAUDE.md 文件控制在 **200 行以内**。文件过长会消耗更多上下文窗口，且降低关键指令的遵循效果。

若内容庞大，可通过以下方式解决：
- **路径限定规则**（`.claude/rules/*.md`）：让某些指令只在处理特定文件时加载
- **路径导入机制**（`@`）：提升组织清晰度（但注意，导入文件仍会在启动时加载，不减少上下文消耗）

### （2）使用清晰结构

建议使用 Markdown 标题和项目符号对相关指令进行分组。结构清晰的章节更容易被 Claude Code 理解和遵循。

推荐组织方式：
```markdown
# 项目概览

# 构建和测试命令

# 代码风格

# Git 工作流

# 重要约束
```

### （3）保持具体、可验证

指令应当尽量具体，避免模糊描述。

| 推荐写法 | 不推荐写法 |
|---------|-----------|
| 使用 2 个空格缩进 | 正确格式化代码 |
| 提交前运行 npm test | 测试你的改动 |
| API 处理器位于 src/api/handlers/ | 保持文件格式良好 |

### （4）避免规则冲突

若 CLAUDE.md 中存在相互矛盾的规则，Claude Code 可能会任意选择其中一条执行。应定期检查：

- 根目录下的 `CLAUDE.md`
- 子目录下嵌套的 `CLAUDE.md`
- `.claude/rules/` 中的规则文件

及时删除过期、重复、冲突的内容。在 monorepo 中，可使用 **claudeMdExcludes** 排除不相关文件。

## 6、导入路径

CLAUDE.md 支持使用 `@path/to/import` 语法导入其他文件。被导入的文件会被展开，并在会话启动时与 CLAUDE.md 一起加载到上下文中。

可导入的内容包括：README、package.json、Git 工作流说明、团队开发规范、个人偏好设置等。

```markdown
请参阅 @README 了解项目概览，并参阅 @package.json 查看项目可用的 npm 命令。

# 补充说明

- Git 工作流 @docs/git-instructions.md
```

- 相对路径：相对于**包含导入语句的文件**解析
- 绝对路径：指向系统中的固定文件（跨项目不建议）
- 被导入文件还可以继续导入其他文件，最多支持**五层**

## 7、CLAUDE.local.md

如果某些指令只适合你个人使用，不想提交到 Git 版本控制，可以在项目根目录创建 `CLAUDE.local.md`，并加入 `.gitignore`。

适合放入的内容：个人常用命令、本地开发环境差异、个人偏好的调试方式、不适合团队共享的临时说明。

> 被 Git 忽略的 `CLAUDE.local.md` 只会存在于创建它的那个 worktree，不会自动同步到其他 worktree。若需多 worktree 共享，可从用户家目录导入。

## 8、AGENTS.md 兼容

Claude Code 默认读取 `CLAUDE.md`，而非 `AGENTS.md`。如果仓库已为其他编码代理维护了 `AGENTS.md`，不需要重复写一份。可以创建 `CLAUDE.md` 导入 `AGENTS.md`，让多个工具共享同一套基础指令：

```markdown
@AGENTS.md

## Claude Code

Use plan mode for changes under `src/billing/`.
```

## 9、使用 .claude/rules/ 整理规则

### （1）为什么使用

当项目不断扩大时，所有规范堆积在一个 CLAUDE.md 文件里会出现问题：内容越来越长、不同主题混杂、局部规则被无差别加载。

`.claude/rules/` 的作用是把规则拆分为独立文件，按主题组织，同时支持**按文件路径生效**——只有当 Claude 处理匹配路径的文件时，对应规则才会加载。

### （2）基本设置

把 Markdown 文件放到 `.claude/rules/` 目录中即可。每个文件最好只覆盖一个主题：

```shell
your-project/
├── .claude/
│   ├── CLAUDE.md           # 主项目指令
│   └── rules/
│       ├── code-style.md   # 代码风格指南
│       ├── testing.md      # 测试约定
│       └── security.md     # 安全要求
```

### （3）无条件加载的规则

没有写 `paths` YAML frontmatter 的规则文件会在启动时直接加载，优先级与 CLAUDE.md 相同。适合放置全局规范：通用代码风格、提交前测试要求、安全基线要求等。

### （4）路径特定规则

若某些规则只适用于特定目录或文件类型，可在规则文件顶部使用 YAML frontmatter 的 `paths` 字段限定：

```yaml
---
paths:
  - "src/api/**/*.ts"
---

# API 开发规则

- 所有 API 接口都必须包含输入校验
- 使用统一的错误响应格式
- 补充 OpenAPI 文档注释
```

`paths` 支持 glob 模式，常见示例：

| 模式 | 含义 |
|------|------|
| `**/*.ts` | 匹配任意目录下的 TypeScript 文件 |
| `src/**/*` | 匹配 `src/` 目录下的所有文件 |
| `*.md` | 匹配项目根目录中的所有 Markdown 文件 |
| `src/**/*.{ts,tsx}` | 匹配多个扩展名 |

### （5）项目级规则

除了项目内的 `.claude/rules/`，还可以在用户主目录下维护个人规则：

```shell
~/.claude/rules/
├── preferences.md    # 个人编码偏好
└── workflows.md      # 个人工作流习惯
```

**用户级规则会先于项目规则加载**，若二者冲突，通常以**项目规则优先**。

## 10、大型团队管理

### （1）部署组织级 CLAUDE.md

组织级 CLAUDE.md 放置在系统指定的托管策略路径中，为所有开发者提供统一的行为指引。**组织级 CLAUDE.md 不能被个人设置排除**。

### （2）排除特定 CLAUDE.md 文件

在大型 monorepo 中，可使用 `claudeMdExcludes` 设置按路径或 glob 模式排除特定的 CLAUDE.md 文件或规则目录：

```json
{
  "claudeMdExcludes": [
    "**/monorepo/CLAUDE.md",
    "/home/user/monorepo/other-team/.claude/rules/**"
  ]
}
```

> `claudeMdExcludes` 使用 glob 语法，针对**绝对文件路径**匹配。托管策略中的 CLAUDE.md 不能被排除。

# 三、自动记忆（Auto Memory）

## 1、概念

自动记忆（Auto Memory）可以让 Claude Code 在不需要手动维护的情况下，**跨会话积累项目相关知识**。它会在工作过程中，根据内容是否值得长期保留，自行决定是否记录一些信息，例如构建命令、调试经验、架构说明、代码风格偏好和工作流习惯。

> Claude Code 不会在每次会话后都写入记忆，而是只保存那些对未来对话可能有帮助的内容。

## 2、启用和禁用

自动记忆默认是**开启**的。可通过两种方式控制：

| 方式 | 操作 |
|------|------|
| 界面开关 | 在会话中输入 `/memory`，通过界面里的自动记忆开关启用或关闭 |
| 配置项 | 在项目设置中设置 `autoMemoryEnabled` |
| 环境变量 | `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1` |

关闭自动记忆的配置示例：
```json
{
  "autoMemoryEnabled": false
}
```

## 3、存储位置

每个项目都有独立的自动记忆目录，默认位置是：

```bash
~/.claude/projects/<project>/memory/
```

其中 `<project>` 通常根据 Git 仓库路径推导得到。因此：
- 同一个仓库的所有 worktree
- 同一个仓库的所有子目录

默认共享同一个自动记忆目录。若当前目录未被 Git 纳入版本控制，则使用项目根目录确定记忆位置。

若希望改存到其他地方，可在**用户设置**或**本地设置**中配置 `autoMemoryDirectory`：

```json
{
  "autoMemoryDirectory": "~/my-custom-memory-dir"
}
```

> 此配置可来自 policy 设置、local 设置、user 设置，但**不能来自项目设置**（避免共享项目把自动记忆重定向到不安全位置）。

## 4、目录结构

自动记忆目录中通常包含一个入口文件和若干主题文件：

```shell
~/.claude/projects/<project>/memory/
├── MEMORY.md
├── debugging.md
├── api-conventions.md
└── ...
```

| 文件 | 作用 |
|------|------|
| `MEMORY.md` | **索引文件**，记录记忆内容的组织形式 |
| `debugging.md`、`api-conventions.md` 等 | 按主题拆分的详细笔记 |

Claude Code 会根据需要继续创建其他主题文件。

## 5、工作方式

每次对话开始前，Claude Code **不会**把整个记忆目录全部读入上下文，而是只加载 `MEMORY.md` 的一部分：

- 前 **200 行**
- 或前 **25 KB**
- 取两者中较小者

超过这个范围的内容不会在启动时自动加载。为了保持启动上下文精简，Claude Code 会尽量把详细信息拆到单独的主题文件中，而让 `MEMORY.md` 保持简洁。

像 `debugging.md`、`patterns.md` 这类主题文件**不会在会话开始时自动加载**。只有当 Claude Code 认为相关信息有用时，才会按需读取这些文件。

在会话过程中，Claude 也会持续读写这些记忆文件。如果在界面中看到 `Writing memory` 或 `Recalled memory`，即表示正在操作记忆文件。

## 6、审计与编辑

自动记忆文件本质上就是普通的 Markdown 文件，因此你可以随时：

- 手动查看
- 直接编辑
- 删除不想保留的内容

如果想在会话中查看这些文件，可以使用 `/memory`。

# 四、对比与常见问题

## 1、CLAUDE.md 与 Auto Memory 对比

| 维度 | CLAUDE.md | Auto Memory |
|------|-----------|-------------|
| **维护方式** | 用户手动编写 | Claude Code 自动生成 |
| **定位** | 事先约定 | 在使用中学习 |
| **适合内容** | 长期、明确、需主动约束行为的规则 | 逐步形成的知识、偏好、经验 |
| **典型示例** | 团队规范、目录约定、测试流程 | 常见构建命令、调试经验、个人偏好 |
| **文件格式** | 标准 Markdown | 标准 Markdown |
| **加载策略** | 启动时全部加载（向上查找拼接） | 只加载 MEMORY.md 前 200 行/25KB |
| **层级支持** | 组织/项目/用户/本地四级 | 项目级（可按项目配置目录） |

## 2、不同工具的实现对比

| 特性 | Claude Code | Cursor | GitHub Copilot |
|------|-------------|--------|----------------|
| 手动记忆 | `CLAUDE.md` | `.cursorrules` | 通过 VS Code 设置 |
| 自动记忆 | `Auto Memory` | 无（需手动维护） | 无 |
| 路径限定规则 | `.claude/rules/*.md` | 无 | 无 |
| 层级加载 | 向上查找 + 拼接 | 单文件 | 编辑器配置 |

## 3、常见问题排查

### （1）Claude Code 没有遵循 CLAUDE.md

需要明确：CLAUDE.md 的内容并不是系统提示本身的一部分，而是作为一条额外的用户消息传递给模型。Claude Code 会尽量遵循，但不等同于硬性强制执行。

排查顺序：
1. 运行 `/memory`，确认当前会话中确实加载了 CLAUDE.md
2. 检查 CLAUDE.md 是否放在了当前会话能加载的位置
3. 尽量把指令写得具体、明确、可验证
4. 检查不同的 CLAUDE.md 文件之间是否存在冲突

> 若某条指令必须提升到更强优先级，可考虑使用 `--append-system-prompt`（适合脚本或自动化场景）。

### （2）不知道自动记忆保存了什么

直接运行 `/memory`，然后打开自动记忆文件夹查看。

### （3）CLAUDE.md 太大

- 使用路径限定规则，让某些指令只在处理特定文件时加载
- 删除那些并不需要在每次会话中都出现的内容
- 把全局规则、局部规则和任务说明拆开管理

> 把内容拆成 `@path` 导入虽然有助于组织结构更清晰，但**不会减少上下文占用**。

### （4）`/compact` 之后指令似乎丢失了

- 项目根目录下的 `CLAUDE.md` 会在压缩后继续保留（Claude Code 会重新从磁盘读取并注入）
- 子目录中的嵌套 `CLAUDE.md` **不会自动重新注入**，只有在后续再次读取对应子目录文件时才会重新加载

> 若希望某条规则在压缩后依然稳定保留，最可靠的方法是写进 `CLAUDE.md`，而不是只停留在对话内容里。

# 五、补充

## 1、记忆系统的通用设计原则

1. **上下文 vs 配置**：记忆本质上是"上下文"而非"强制配置"，效果取决于内容清晰度
2. **分层隔离**：组织/项目/个人/本地四级隔离，避免个人偏好污染团队规范
3. **路径限定**：复杂项目应使用路径限定规则，让特定规则只在处理匹配文件时加载
4. **审计友好**：记忆文件是标准 Markdown，可随时查看、编辑、删除

## 2、CLAUDE.md 注释

在 CLAUDE.md 中可以使用块级 HTML 注释，用于给人工维护者留下说明：

```HTML
<!-- 注释 -->
```

这些注释在注入到 Claude Code 的上下文时会被移除，因此不会消耗上下文 token。

> 代码块内部的注释仍然会保留。Claude Code 使用 Read 工具直接打开 CLAUDE.md 文件时，这些注释仍然可见。

## 3、使用符号链接共享规则

`.claude/rules` 目录支持符号链接，可以把一套公共规则维护在统一位置，再链接到多个项目复用：

```shell
ln -s ~/shared-claude-rules .claude/rules/shared
ln -s ~/company-standards/security.md .claude/rules/security.md
```
