# 一、概述

**子代理（Subagent）** 是 Claude Code 中专门处理特定任务的 AI 助手，拥有独立的上下文窗口、自定义系统提示词和特定的工具权限。当主代理判断任务符合某个子代理的描述时，会自动将任务委托给它执行。

子代理机制是 Claude Code 区别于普通 AI 对话工具的核心能力之一，它通过任务隔离、权限控制和专业化分工，大幅提升了复杂开发场景下的效率与安全性。

> 核心原则：留在主对话做需要频繁沟通的小任务，使用子代理处理独立、高输出、需要权限限制的任务。

---

# 二、核心优势

## 1、上下文保护

将探索和实现过程隔离到独立上下文，避免污染主对话。主对话只接收子代理返回的摘要或结果，不会因中间过程的冗长输出而丢失关键上下文。

## 2、权限隔离

通过限制可用工具强制执行约束。例如可以创建一个只读审查器子代理，它只能使用 Read、Glob、Grep 等读取类工具，从根本上杜绝误修改代码的风险。

## 3、专业化

每个子代理针对特定领域深度定制。可以为代码审查、测试编写、文档生成、安全审计等不同场景配置专门的子代理，每个都配备最适合的系统提示词和工具集。

## 4、成本控制

简单任务可路由到更快、更便宜的模型（如 Haiku），复杂任务使用更强的模型（如 Opus），实现按需分配计算资源。

## 5、跨项目复用

用户级子代理（存放在 `~/.claude/agents/`）可在多个项目中共享，一次配置到处使用。

# 三、常见使用模式

## 1、隔离高输出量操作

将产生大量中间内容的任务委托给子代理，主对话只接收摘要：

| 场景 | 子代理任务 | 主对话接收 |
|------|-----------|-----------|
| 运行测试套件 | 执行全部测试并分析失败原因 | 测试摘要与修复建议 |
| 分析日志文件 | 扫描大型日志定位异常 | 异常汇总与根因分析 |
| 扫描大型代码库 | 全局搜索特定模式 | 匹配结果列表 |
| 获取大量文档 | 批量读取 API 文档 | 关键信息提炼 |

> 此模式的核心价值在于：主对话的上下文窗口不会被海量中间输出占满。

## 2、并行研究

多个调查任务彼此独立时，同时启动多个子代理并行处理：

```
使用不同的子代理并行研究认证模块、数据库模块和 API 模块
```

三个子代理同时工作，各自探索不同模块，最后将结果汇总到主对话进行整合分析。相比串行执行，效率提升数倍。

## 3、串联工作流

多步骤任务按顺序调用多个子代理，形成流水线：

```
先使用 code-reviewer 查找性能问题，然后使用 optimizer 修复
```

前一个子代理的输出作为后一个子代理的输入，实现分析 → 规划 → 执行的自动化流水线。

# 四、子代理配置

## 1、文件格式

子代理文件使用 **YAML frontmatter + Markdown system prompt** 格式：

```yaml
---
name: code-review
description: 资深代码审查专家。在代码更改后主动使用。
tools: Read, Glob, Grep, Bash
model: sonnet
permissionMode: default
memory: project
---

你是一名代码审查员。被调用时，请分析代码，并针对代码质量、安全性和最佳实践提供具体、可执行的反馈。
```

## 2、关键配置字段

| 字段 | 说明 | 示例值 |
|------|------|--------|
| `name` | 唯一标识符（小写字母和连字符） | `code-review` |
| `description` | 触发条件描述 | "在代码更改后主动使用" |
| `tools` | 白名单工具列表 | `Read, Glob, Grep` |
| `disallowedTools` | 黑名单工具列表 | `Write, Edit` |
| `model` | 使用的模型 | `sonnet` / `opus` / `haiku` / `inherit` |
| `permissionMode` | 权限模式 | 可覆盖主会话设置 |
| `mcpServers` | 可用的 MCP 服务器 | 可内联定义或引用 |
| `skills` | 预加载的技能 | 完整内容注入上下文 |
| `memory` | 持久记忆作用域 | `user` / `project` / `local` |
| `isolation` | 隔离模式 | `worktree` 时在隔离 git worktree 中运行 |
| `background` | 后台运行 | `true` 时作为后台任务运行 |
| `maxTurns` | 最大轮数 | 限制子代理的对话轮次 |
| `effort` | 努力程度 | 控制子代理的推理深度 |
| `color` | 显示颜色 | 在 UI 中标识该子代理 |
| `initialPrompt` | 初始提示词 | 启动时自动发送的额外指令 |

> `description` 字段至关重要，它决定了主代理在什么情况下会将任务委托给该子代理。描述应清晰说明触发条件和职责范围。

## 3、作用域与优先级

当多个位置定义了同名子代理时，按优先级生效：

| 优先级 | 位置 | 范围 |
|--------|------|------|
| 1 | 托管设置 | 组织范围 |
| 2 | `--agents` CLI 标志 | 当前会话 |
| 3 | `.claude/agents/` | 当前项目 |
| 4 | `~/.claude/agents` | 个人所有项目 |
| 5 | 插件 | 启用该插件的位置 |

> 高优先级配置会覆盖低优先级配置。项目级子代理适合团队协作，用户级子代理适合个人习惯。

# 五、内置子代理类型

Claude Code 内置了若干常用子代理：

| 子代理 | 模型 | 工具权限 | 用途 |
|--------|------|----------|------|
| `Explore` | Haiku | 只读（Read, Glob, Grep 等） | 代码库探索 |
| `Plan` | Sonnet | 只读 | 规划模式研究 |
| `General-purpose` | 继承主会话 | 所有工具 | 通用复杂任务 |
| `statusline-setup` | - | 辅助工具 | 状态栏配置 |
| `Claude Code Guide` | - | 辅助工具 | 使用指南 |

> 内置子代理无需配置即可使用，适合快速上手。自定义子代理则可以根据团队需求深度定制。

# 六、使用策略：主对话 vs 子代理

## 1、留在主对话的场景

- 需要频繁沟通、调整方向的任务
- 上下文连续的小任务（修改一两个文件）
- 需要即时反馈和迭代确认的交互

## 2、使用子代理的场景

- 独立任务（与当前对话主线无关）
- 大量中间输出（测试、日志、扫描）
- 需要限制工具权限（只读审查、安全审计）
- 专业角色处理（代码审查、性能优化、文档生成）
- 需要并行处理的多项独立研究

| 维度 | 主对话 | 子代理 |
|------|--------|--------|
| 上下文连续性 | 高 | 独立隔离 |
| 工具权限 | 继承主会话 | 可独立配置 |
| 输出量 | 受限于主窗口 | 可大量产生 |
| 并行能力 | 串行 | 可并行启动多个 |
| 适用模型 | 继承主会话 | 可单独指定 |

# 七、补充

## 1、创建子代理的步骤

1. 在项目目录创建 `.claude/agents/` 文件夹（或用户目录 `~/.claude/agents/`）
2. 新建 `.md` 文件，文件名即子代理标识
3. 编写 YAML frontmatter 定义配置
4. 编写 Markdown 内容作为系统提示词
5. 在对话中通过描述触发，或显式调用

## 2、最佳实践

- **描述要精准**：`description` 直接决定触发命中率，应明确说明"在什么情况下使用我"
- **权限最小化**：只授予完成任务必需的工具，降低误操作风险
- **模型按需选择**：探索类任务用 Haiku 更快更便宜，复杂分析用 Opus
- **善用并行**：多个独立研究任务同时启动子代理，显著缩短总耗时
- **结果摘要化**：子代理应在返回前自行总结关键发现，避免向主对话倾倒原始输出
