# 一、概述

在 Claude Code 的交互式会话中，**子代理（Subagent）** 是专门处理特定任务的 AI 助手。每个子代理拥有独立的上下文窗口、自定义系统提示词和特定的工具权限。当主代理判断任务符合某个子代理的描述时，会自动将任务委托给它。

子代理的本质是**任务委托与上下文隔离机制**。它让主对话保持精简，同时将复杂、高输出量或需要专业角色的工作交给专门的处理单元。

> 子代理不是简单的"调用外部工具"，而是拥有完整推理能力的独立会话。主代理只接收子代理的摘要结果，中间过程不会污染主对话上下文。


# 二、核心优势

## 1、上下文保护

将探索和实现过程隔离到独立上下文，避免大量中间内容污染主对话。例如：
- 运行测试套件产生的日志
- 扫描大型代码库的遍历结果
- 分析复杂日志文件的中间推理

## 2、权限隔离

通过限制可用工具强制执行约束。典型场景：
- 创建**只读审查器**，禁止任何文件修改操作
- 创建**安全审计器**，仅开放 Read 和 Grep 工具
- 创建**文档生成器**，只允许写入 docs/ 目录

## 3、专业化

每个子代理针对特定领域深度定制系统提示词，例如：
- `code-reviewer`：专注于代码质量、安全性和最佳实践
- `optimizer`：专注于性能分析和优化建议
- `test-writer`：专注于测试用例生成和覆盖率

## 4、成本控制

简单任务可路由到更快、更便宜的模型（如 Haiku），复杂任务使用更强的模型（如 Opus）。

## 5、跨项目复用

用户级子代理（存放在 `~/.claude/agents/`）可在多个项目中共享，一次配置，到处使用。


# 三、常见使用模式

## 1、隔离高输出量操作

将产生大量中间内容的任务委托给子代理，主对话只接收摘要：

| 场景 | 委托内容 | 主对话接收 |
|------|----------|------------|
| 运行测试套件 | 完整测试输出、失败详情 | 测试摘要、失败列表 |
| 分析日志文件 | 日志逐行分析过程 | 异常模式总结 |
| 扫描大型代码库 | 文件遍历和代码阅读 | 架构概览和关键发现 |
| 获取大量文档 | 文档检索和解析 | 核心要点提炼 |

> 典型指令："请使用 code-explorer 子代理扫描 src/ 目录，给我一份架构概览。"

## 2、并行研究

多个调查任务彼此独立时，同时启动多个子代理并行处理：

```
使用不同的子代理并行研究：
- auth-agent → 认证模块的安全审计
- db-agent → 数据库层的性能瓶颈
- api-agent → API 接口的文档完整性
```

## 3、串联工作流

多步骤任务按顺序调用多个子代理，形成流水线：

```
步骤1: code-reviewer 查找性能问题
    ↓
步骤2: optimizer 生成修复方案
    ↓
步骤3: test-writer 编写回归测试
    ↓
步骤4: code-reviewer 验证修复质量
```


# 四、子代理配置

## 1、文件格式

子代理文件使用 **YAML frontmatter + Markdown system prompt** 格式：

```yaml
---
name: code-review
description: 资深代码审查专家。在代码更改后主动使用。
tools: Read, Glob, Grep, Bash
model: sonnet
permissionMode: default
memory: project
---

你是一名代码审查员。被调用时，请分析代码，并针对代码质量、安全性和最佳实践提供具体、可执行的反馈。
```

## 2、关键配置字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | string | 唯一标识符（小写字母和连字符） |
| `description` | string | 触发条件描述，决定主代理何时委托任务 |
| `tools` | list | 白名单工具列表，未列出的工具不可用 |
| `disallowedTools` | list | 黑名单工具列表，与 `tools` 互斥 |
| `model` | string | 使用的模型：`sonnet` / `opus` / `haiku` / `inherit` |
| `permissionMode` | string | 权限模式，可覆盖主会话设置 |
| `mcpServers` | list | 可用的 MCP 服务器（可内联定义或引用） |
| `skills` | list | 预加载的技能（完整内容注入上下文） |
| `memory` | string | 持久记忆作用域：`user` / `project` / `local` |
| `isolation` | string | 设置为 `worktree` 时在隔离 git worktree 中运行 |
| `background` | boolean | 设置为 `true` 时作为后台任务运行 |
| `maxTurns` | number | 最大对话轮数限制 |
| `effort` | string | 思考强度：`low` / `medium` / `high` / `max` |
| `color` | string | 终端显示颜色 |
| `initialPrompt` | string | 启动时自动发送的初始提示 |

## 3、内置子代理

Claude Code 内置了若干常用子代理：

| 子代理 | 功能 | 特点 |
|--------|------|------|
| `Explore` | 只读代码库探索 | 使用 Haiku 模型，轻量快速 |
| `Plan` | 规划模式研究 | 只读，用于大型改造前的方案设计 |
| `General-purpose` | 通用复杂任务 | 拥有所有工具权限 |
| `statusline-setup` | 状态栏配置辅助 | 帮助配置底部状态栏 |
| `Claude Code Guide` | 使用指南 | 解答 Claude Code 使用问题 |


# 五、作用域与优先级

当多个位置定义了同名子代理时，按优先级生效：

| 优先级 | 位置 | 范围 | 说明 |
|--------|------|------|------|
| 1 | 托管设置 | 组织范围 | IT/DevOps 统一管理 |
| 2 | `--agents` CLI 标志 | 当前会话 | 临时指定代理目录 |
| 3 | `.claude/agents/` | 当前项目 | 项目专属子代理 |
| 4 | `~/.claude/agents` | 个人所有项目 | 用户级共享子代理 |
| 5 | 插件 | 启用该插件的位置 | 插件提供的子代理 |

> 高优先级会覆盖低优先级。例如，项目级 `.claude/agents/` 中的同名子代理会覆盖用户级 `~/.claude/agents/` 中的定义。


# 六、与主对话的选择策略

## 1、留在主对话的场景

- 需要频繁沟通、调整方向的小任务
- 上下文连续性要求高（如逐步迭代同一功能）
- 用户希望实时观察推理过程

## 2、使用子代理的场景

- 独立任务，不需要中途干预
- 大量中间输出，可能撑满主对话上下文
- 需要限制工具权限（如只读审查）
- 需要专业角色处理（如安全审计、性能优化）
- 需要并行处理多个独立任务

> 判断原则：**如果任务完成后你只需要一个摘要结果，就使用子代理；如果你需要边做边调整，就留在主对话。**


# 七、典型配置示例

## 1、只读代码审查器

```yaml
---
name: code-reviewer
description: 资深代码审查专家。在代码更改后主动使用，只提供反馈不修改代码。
tools: Read, Glob, Grep
disallowedTools: Write, Edit, Bash
model: sonnet
permissionMode: plan
---

你是一名严格的代码审查员。被调用时：
1. 分析相关代码变更
2. 针对代码质量、安全性和最佳实践提供具体、可执行的反馈
3. 使用行号引用具体问题位置
4. 绝不修改任何文件
```

## 2、测试驱动开发助手

```yaml
---
name: tdd-helper
description: 测试驱动开发助手。在需要编写测试或执行红-绿-重构循环时使用。
tools: Read, Write, Edit, Bash
model: sonnet
memory: project
---

你是一名 TDD 实践者。遵循以下流程：
1. 先编写失败的测试（红）
2. 运行测试确认失败
3. 编写最少代码使测试通过（绿）
4. 运行测试确认通过
5. 重构代码，保持测试通过
6. 重复上述循环
```

## 3、文档生成器

```yaml
---
name: doc-generator
description: API 文档生成专家。在需要生成或更新 API 文档时使用。
tools: Read, Write
model: haiku
---

你是一名技术文档工程师。被调用时：
1. 阅读相关代码和接口定义
2. 生成清晰、准确的 API 文档
3. 包含参数说明、返回值、异常情况和示例
4. 只写入 docs/ 目录下的文件
```


# 八、Superpowers 框架中的子代理

Superpowers 框架大量依赖子代理驱动开发流程。在其 7 阶段标准化工作流中，子代理扮演核心角色：

| 阶段 | 子代理作用 |
|------|-----------|
| 头脑风暴 | 苏格拉底式提问，完善设计想法 |
| 编写计划 | 将工作分解为小块任务（每个 2-5 分钟） |
| 执行计划 | 为每个任务派遣独立子代理 |
| 代码审查 | 两阶段审查（规格合规性 + 代码质量） |
| TDD 循环 | 子代理执行红-绿-重构 |
| 并行代理 | 并发子代理工作流加速开发 |

> Superpowers 的核心理念是"系统性优于临时性"，子代理正是这种系统化的执行单元。


# 九、常见问题

## 1、子代理修改了文件，主对话能看到吗？

可以。子代理的文件修改会反映在工作目录中，主对话可以通过 Read 工具或 `/diff` 查看变更。但子代理的**中间推理过程**不会进入主对话上下文。

## 2、子代理能访问主对话的上下文吗？

不能。子代理拥有独立的上下文窗口，启动时只接收主代理传递的任务描述。如果需要特定上下文，主代理需要在委托时明确传递。

## 3、如何管理大量子代理？

建议按层级组织：
- `~/.claude/agents/`：通用型子代理（跨项目复用）
- `.claude/agents/`：项目专属子代理
- 使用清晰的命名约定：`领域-角色`，如 `backend-reviewer`、`frontend-optimizer`

## 4、子代理失败怎么办？

子代理的错误会返回给主代理。主代理可以：
- 重新委托（调整提示词或参数）
- 自己接管处理
- 将任务拆分为更小的子任务分别委托
